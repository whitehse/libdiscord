# ADR 004: Dialectic Client-Server Testing

**Date**: 2026-06-29
**Status**: Accepted
**Deciders**: Project maintainers

## Context
libdiscord sits on top of librest, which sits on top of shaggy. Testing the full stack requires verifying that HTTP/2 frames generated by the Discord REST helpers are correctly interpreted by the receiving side, and that Gateway WebSocket frames round-trip correctly.

## Decision
We use **dialectic testing**: paired client↔server contexts exchanging buffers directly in memory, without any networking.

Pattern:
1. Create client and server `discord_ctx_t` with matching configs.
2. Perform HTTP/2 handshake (exchange preface + SETTINGS via buffer swap).
3. Client calls a Discord API helper (e.g., `discord_create_message`).
4. Extract HTTP/2 frames from client output.
5. Feed frames to server input.
6. Verify server-side events.
7. Server sends response frames.
8. Feed response to client.
9. Verify client-side events (e.g., `DISCORD_EVENT_API_RESPONSE`).

For Gateway testing:
1. Client calls `discord_gateway_connect` (establishes binary channel).
2. Exchange frames.
3. Client sends identify/heartbeat.
4. Verify server receives WebSocket frames.

## Consequences
- No sockets required — tests run in milliseconds.
- Full round-trip verification without network dependencies.
- Tests serve as executable documentation of the API contract.
- Any new feature must be exercised via a dialectic test.

## Verification
- `test_discord_dialectic.c` contains REST and Gateway round-trip tests.
